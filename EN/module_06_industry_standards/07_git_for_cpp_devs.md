## 1. Basic Git Workflow

### 1.1 Creating a Repository

```bash
# EN: Initialize a new repository
git init my_cpp_project
cd my_cpp_project

# EN: Or clone an existing one
git clone https://github.com/user/project.git
git clone --depth 1 https://github.com/user/project.git  # Shallow clone (faster)
```

### 1.2 Basic Cycle

```
  [Working Directory] --git add--> [Staging Area] --git commit--> [Local Repo] --git push--> [Remote]
       ↑                                                              |
       └──────────────────── git checkout / reset ────────────────────┘
```

```bash
# EN: Stage specific files
git add src/main.cpp src/engine.cpp

# EN: Stage all changes
git add .

# EN: Commit with descriptive message
git commit -m "feat: add CAN bus parser for ECU diagnostics"

# EN: Push to remote
git push origin main

# EN: Pull latest changes
git pull origin main
```

### 1.3 Commit Message Convention

```
# EN: Use Conventional Commits format (widely adopted in industry):

<type>(<scope>): <description>

Types:
  feat:     New feature
  fix:      Bug fix
  refactor: Code refactoring
  test:     Adding/updating tests
  docs:     Documentation
  perf:     Performance improvement
  build:    Build system changes
  ci:       CI/CD changes

# Good examples
git commit -m "feat(can): implement ISO-TP multi-frame reception"
git commit -m "fix(memory): resolve double-free in sensor buffer"
git commit -m "perf(simd): vectorize signal processing with AVX2"
git commit -m "refactor(ecu): extract diagnostic handler to separate class"
```

## 3. Branch Stratejisi / Branching Strategy

### 3.1 Feature Branch Workflow

```
main (or master)
 │
 ├── develop                  ← Integration branch (optional)
 │    ├── feature/can-parser   ← Feature branches
 │    ├── feature/ecu-diag
 │    └── feature/simd-opt
 │
 ├── release/v2.1             ← Release preparation
 │
 └── hotfix/memory-leak       ← Emergency production fixes
```

```bash
# EN: Create a feature branch
git checkout -b feature/can-parser

# EN: Work on your feature (multiple commits are fine)
git add src/can_parser.cpp include/can_parser.hpp
git commit -m "feat(can): add basic CAN frame parsing"

git add src/can_parser.cpp tests/test_can_parser.cpp
git commit -m "feat(can): add extended frame (29-bit ID) support"

# EN: Push feature branch to remote
git push -u origin feature/can-parser

# EN: When done, merge back to main (or create PR)
git checkout main
git merge --no-ff feature/can-parser    # --no-ff creates a merge commit
git branch -d feature/can-parser         # Delete the feature branch
```

### 3.2 Merge vs Rebase

```bash
# EN: MERGE: Creates a merge commit, preserves full branch history
#     Good for: Feature branches merged via PR, keeping context

git checkout main
git merge feature/can-parser

# EN: REBASE: Replays your commits on top of main, linear history
#     Good for: Cleaning up local work before merging, bisect-friendly
#     WARNING: Never rebase commits that are already pushed and shared!

git checkout feature/can-parser
git rebase main
git checkout main
git merge feature/can-parser    # Fast-forward merge after rebase
```

**Golden Rule:**  
> EN: **Never rebase commits that have been pushed to a shared branch.**

## 4. Conflict Resolution

```bash
# EN: When a merge fails due to conflicts:

git merge feature/can-parser
# Auto-merging src/can_parser.cpp
# CONFLICT (content): Merge conflict in src/can_parser.cpp
# Automatic merge failed; fix conflicts and then commit the result.

git status   # See which files have conflicts

# EN: Open the conflicted file — you'll see markers:
```

```cpp
// EN: Conflict markers in the file:
<<<<<<< HEAD
void parse_frame(const CANFrame& frame) {
    // Current branch's version (HEAD = main)
=======
void parse_frame(CANFrame frame) {
    // Incoming branch's version (feature/can-parser)
>>>>>>> feature/can-parser
```

```bash
# EN: After manually resolving:
git add src/can_parser.cpp       # Mark as resolved
git commit                       # Complete the merge (default message is fine)
```

**Conflict Prevention Tips:**
1. **Pull frequently:** `git pull --rebase origin main`
2. **Small commits:** Less conflict risk
3. **Header/source separation:** `.hpp` interface, `.cpp` implementation
4. **Forward declarations:** Reduce `#include` dependencies
5. **clang-format:** Same formatting = no whitespace conflicts

## 5. Git Hooks

```bash
# EN: Git hooks are scripts that run automatically at certain points.
#     Stored in .git/hooks/ (local) or managed via tools like pre-commit.
```

### 5.1 pre-commit: clang-format

```bash
#!/bin/bash
# .git/hooks/pre-commit — Auto-format C++ files before commit
# EN: This hook runs clang-format on all staged .cpp/.hpp files

STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(cpp|hpp|h|cc|cxx)$')

if [ -n "$STAGED_FILES" ]; then
    echo "Running clang-format on staged C++ files..."
    for FILE in $STAGED_FILES; do
        clang-format -i "$FILE"
        git add "$FILE"      # Re-stage formatted file
    done
fi
```

### 5.2 pre-commit: clang-tidy

```bash
#!/bin/bash
# .git/hooks/pre-commit — Static analysis check
# EN: Runs clang-tidy on changed files and blocks commit if issues found

STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(cpp|cc|cxx)$')

if [ -n "$STAGED_FILES" ]; then
    echo "Running clang-tidy..."
    ERRORS=0
    for FILE in $STAGED_FILES; do
        if ! clang-tidy "$FILE" -- -std=c++17 2>/dev/null; then
            ERRORS=1
        fi
    done
    if [ "$ERRORS" -ne 0 ]; then
        echo "ERROR: clang-tidy found issues. Fix them before committing."
        exit 1
    fi
fi
```

### 5.3 pre-push: Build & Test

```bash
#!/bin/bash
# .git/hooks/pre-push — Build and test before push
# EN: Ensures code compiles and tests pass before pushing

echo "Running build and tests before push..."
cd build && cmake --build . --target all && ctest --output-on-failure
if [ $? -ne 0 ]; then
    echo "ERROR: Build or tests failed. Push aborted."
    exit 1
fi
echo "All checks passed. Pushing..."
```

### 5.4 Modern Alternative: pre-commit Framework

```yaml
# .pre-commit-config.yaml — Industry-standard hook management
# Install: pip install pre-commit && pre-commit install

repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-merge-conflict

  - repo: https://github.com/pocc/pre-commit-hooks  # C++ specific
    rev: v1.3.5
    hooks:
      - id: clang-format
        args: [--style=file]    # Uses .clang-format in project root
      - id: clang-tidy
        args: [-p=build]        # Uses compile_commands.json from build/
      - id: cppcheck
```

## 6. Submodules ve Dependency Management

### 6.1 Git Submodules

```bash
# EN: Add an external library as a submodule
git submodule add https://github.com/google/googletest.git extern/googletest
git submodule add https://github.com/nlohmann/json.git extern/json

# EN: Clone a project with submodules
git clone --recurse-submodules https://github.com/user/project.git

# EN: If already cloned, initialize submodules
git submodule update --init --recursive

# EN: Update submodule to latest upstream version
cd extern/googletest
git checkout main && git pull
cd ../..
git add extern/googletest
git commit -m "build: update googletest to latest"
```

```cmake
# CMakeLists.txt — Using submodule in CMake
add_subdirectory(extern/googletest)
target_link_libraries(my_tests PRIVATE gtest_main)
```

### 6.2 Alternatives to Submodules

```
┌─────────────────┬───────────────────────────────────────────────────────┐
│ Method          │ Pros / Cons                                           │
├─────────────────┼───────────────────────────────────────────────────────┤
│ Git Submodules  │ ✅ Exact version pinning. ❌ Confusing workflow       │
│ FetchContent    │ ✅ CMake-native, auto-download. ❌ Re-downloads often │
│ vcpkg           │ ✅ Large package registry. ❌ Microsoft-centric       │
│ Conan           │ ✅ Python-based, flexible. ❌ Learning curve          │
│ Header-only     │ ✅ Just copy .hpp file. ❌ No version management      │
│ System packages │ ✅ apt/brew install. ❌ Version may differ            │
└─────────────────┴───────────────────────────────────────────────────────┘
```

```cmake
# CMake FetchContent — Modern alternative to submodules
include(FetchContent)
FetchContent_Declare(
    googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.14.0  # Pin to specific version!
)
FetchContent_MakeAvailable(googletest)
```

## 7. Useful Git Commands

```bash
# ─── History & Investigation
git log --oneline --graph --all          # Visual branch history
git log --oneline -20                     # Last 20 commits
git log --author="Murat" --since="1 week ago"  # Filter by author/date
git blame src/engine.cpp                  # Who wrote each line?
git bisect start                          # Binary search for bug-introducing commit

# ─── Stashing
git stash                                 # Save uncommitted work temporarily
git stash pop                             # Restore stashed work
git stash list                            # List all stashes
git stash drop stash@{0}                  # Remove a specific stash

# ─── Undoing Changes
git checkout -- src/main.cpp              # Discard unstaged changes in file
git reset HEAD src/main.cpp               # Unstage a file
git reset --soft HEAD~1                   # Undo last commit, keep changes staged
git revert abc1234                        # Create new commit that undoes abc1234

# ─── Tagging & Releases
git tag v1.0.0                            # Lightweight tag
git tag -a v1.0.0 -m "Release 1.0.0"     # Annotated tag (recommended)
git push origin v1.0.0                    # Push tag to remote
git push origin --tags                    # Push all tags

# ─── Cleaning / Temizlik ────────────────────────────────────────
git clean -fd                             # Remove untracked files and directories
git gc                                    # Garbage collect (compress objects)
```

## 8. C++-Specific Tips

### 8.1 compile_commands.json Tracking

```bash
# EN: compile_commands.json is generated by CMake and used by clangd/clang-tidy.
#     Some teams track it, some don't. Best practice:

# Option 1: Track it (easier for teammates to get IDE support immediately)
# Option 2: Add to .gitignore and regenerate (cleaner, avoids merge conflicts)

# Generate with CMake:
cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
ln -sf build/compile_commands.json .
```

### 8.2 Large Binary Files

```bash
# EN: C++ projects sometimes have large test data, firmware images, etc.
#     Use Git LFS (Large File Storage) for these:

git lfs install
git lfs track "*.bin"         # Firmware images
git lfs track "*.hex"         # Intel HEX files
git lfs track "*.elf"         # ELF binaries
git lfs track "*.dbc"         # CAN database files
git lfs track "*.a2l"         # ASAM MCD-2MC calibration files
git add .gitattributes        # LFS tracking info stored here
```

### 8.3 Useful Git Aliases for C++ Devs

```bash
# Add to ~/.gitconfig:
[alias]
    # Quick status
    s = status -sb

    # Pretty log
    lg = log --oneline --graph --decorate --all -20

    # Show what changed in last commit
    last = log -1 --stat

    # List files changed in a commit
    files = diff-tree --no-commit-id --name-only -r

    # Force-rebuild after checkout (useful for CMake)
    rebuild = !cd build && cmake --build . --clean-first

    # Find all TODO/FIXME in tracked files
    todos = grep -n -i 'TODO\\|FIXME\\|HACK\\|XXX'
```
